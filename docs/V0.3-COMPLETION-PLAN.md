touch so that concurrent PRs do not conflict.

## Coordination rules

These are the rules that keep parallel work mergeable. They are not optional —
each one maps to a concrete failure we hit while building DATA-001 / FND-002.

1. **`prisma/schema.prisma` and `prisma/seed.ts` belong to Lane 1 only.** Lanes 2
   and 3 never edit them. (Two agents editing the schema produced two competing
   migrations that could not both land.)
2. **Branch off the latest `main`; rebase _before_ opening the PR, not after.**
   (PR #11 went stale because it was branched off a pre-merge `main` and would
   have reverted PR #12.)
3. **One issue = one branch = one PR.** Branch name `<epic>-NNN-slug`
   (`sec-001-authjs`, `ai-001-llm-dedup`, `ui-001-boundaries`) per
   `CONTRIBUTING.md`.
4. **`package.json` dependency additions:** call them out in the PR title. They
   conflict trivially, but only if reviewers know to look.
5. **`next.config.ts`** is touched by SEC-003 and DPL-001 — land **SEC-003 first**.
6. **CI must be green before merge.** The required status check is the **`gate`
   job** — re-hardened by PR #57 to fan out into three parallel jobs that `gate`
   depends on: `typecheck` (`tsc --noEmit`), `content-validate` (`check:content`
   + `check:nrta`), and `build` (`npm run build`). `main` is branch-protected on
   `gate`. (The local `npm run check` additionally runs `check:schema`.)

## Migration discipline (Lane 1)

Because migrations are append-only and timestamp-named, two unmerged branches
that both run `prisma migrate dev` create divergent migration directories that
cannot both apply. To avoid it:

- Only Lane 1 generates migrations.
- Generate the migration **last**, immediately before opening the PR, off the
  latest `main`.
- If `main` advanced under you, rebase and **regenerate** the migration rather
  than merging two migration dirs.
